# دليل استخدام نظام الدفاع السيبراني (CDS)

دليل عملي خطوة بخطوة لتثبيت المشروع وتشغيله واستخدام الواجهة وواجهات API.

> للتفاصيل المعمارية راجع [`تقرير_النظام.md`](تقرير_النظام.md)

---

## 1. المتطلبات

| الأداة | الإصدار الموصى به |
|--------|-------------------|
| Python | 3.11 أو أحدث |
| Node.js | 18 أو أحدث |
| npm | يأتي مع Node.js |
| Docker | اختياري — لتشغيل PostgreSQL |
| Git | لاستنساخ المستودع |

**أنظمة التشغيل المدعومة:** Windows، Linux، macOS.

---

## 2. التثبيت

### 2.1 استنساخ المشروع

```bash
git clone <رابط-المستودع>
cd cyber-defense-system
```

### 2.2 إعداد ملف البيئة

```bash
cp .env.example .env
```

عدّل `.env` حسب حاجتك (انظر القسم 3).

### 2.3 تثبيت حزم Python

```bash
pip install -r requirements.txt
```

يُفضّل استخدام بيئة افتراضية:

```bash
python -m venv .venv

# Windows
.venv\Scripts\activate

# Linux / macOS
source .venv/bin/activate

pip install -r requirements.txt
```

### 2.4 تثبيت حزم الواجهة

```bash
cd frontend
npm install
cd ..
```

## 3. إعداد المتغيرات البيئية

افتح ملف `.env` في جذر المشروع:

### 3.1 حسابات الدخول

| المتغير | الافتراضي | الدور |
|---------|-----------|-------|
| `CDS_ADMIN_USER` / `CDS_ADMIN_PASSWORD` | admin / changeme | مدير — كامل الصلاحيات |
| `CDS_ANALYST_USER` / `CDS_ANALYST_PASSWORD` | analyst / analyst123 | محلل |
| `CDS_VIEWER_USER` / `CDS_VIEWER_PASSWORD` | viewer / viewer123 | مشاهدة فقط |

> **مهم:** غيّر كلمات المرور قبل أي استخدام خارج الجهاز المحلي.

### 3.2 قاعدة البيانات

**PostgreSQL (موصى به للإنتاج):**

```env
CDS_DATABASE_URL=postgresql://cds:cds_secret@localhost:5432/cyber_defense
```

**SQLite (تطوير سريع بدون Docker):**

اترك `CDS_DATABASE_URL` فارغاً — يُنشأ الملف تلقائياً في `data/cyber_defense.db`.

### 3.3 إعدادات أخرى شائعة

| المتغير | الوصف | الافتراضي |
|---------|--------|-----------|
| `CDS_SECRET_KEY` | مفتاح تشفير الجلسات | يُولَّد تلقائياً إن تُرك فارغاً |
| `CDS_PORT` | منفذ الـ API | 8080 |
| `CDS_BIND_HOST` | عنوان الربط | 127.0.0.1 |
| `CDS_CORS_ORIGINS` | عناوين الواجهة المسموحة | localhost:5173 |
| `CDS_FIREWALL_ENABLED` | تطبيق حظر IP على نظام التشغيل | true |
| `CDS_ML_SHADOW_MODE` | تفعيل ML الاستشاري | true |

## 4. تشغيل قاعدة البيانات (PostgreSQL)

إذا كان Docker مثبتاً:

```bash
docker compose up -d postgres
```

تحقق من التشغيل:

```bash
docker compose ps
```

بدون Docker: ثبّت PostgreSQL محلياً وأنشئ قاعدة `cyber_defense` مع المستخدم `cds`.

## 5. تشغيل المشروع

تحتاج **نافذتي طرفية** (أو أكثر): واحدة للـ API وأخرى للواجهة.

### 5.1 تشغيل الـ Backend (FastAPI)

من جذر المشروع:

```bash
python -m backend.main
```

أو:

```bash
uvicorn backend.main:app --host 127.0.0.1 --port 8080 --reload
```

عند النجاح:

```
INFO: Uvicorn running on http://127.0.0.1:8080
```

تحقق: افتح http://127.0.0.1:8080/api/health — يجب أن ترى `{"status":"ok",...}`.

### 5.2 تشغيل الواجهة (React)

```bash
cd frontend
npm run dev
```

افتح المتصفح على: **http://localhost:5173**

الواجهة تتصل تلقائياً بالـ API عبر البروكسي المضمّن في Vite.

### 5.3 بناء نسخة إنتاج (اختياري)

```bash
cd frontend
npm run build
```

ثم شغّل الـ API فقط — يخدم ملفات `frontend/dist` تلقائياً:

```bash
cd ..
python -m backend.main
```

افتح: http://127.0.0.1:8080

## 6. تسجيل الدخول

1. افتح http://localhost:5173 (أو منفذ الإنتاج 8080).
2. أدخل اسم المستخدم وكلمة المرور من `.env`.
3. اضغط **دخول**.

تظهر في أسفل صفحة الدخول جدول ببيانات الحسابات الفعلية على جهازك.

### الصلاحيات حسب الدور

| الإجراء | مدير | محلل | مشاهدة |
|---------|:----:|:----:|:------:|
| عرض المقاييس والتحليلات | ✓ | ✓ | ✓ |
| عرض التهديدات والأحداث | ✓ | ✓ | ✓ |
| تحليل كيان | ✓ | ✓ | — |
| إنشاء تقرير أمني | ✓ | ✓ | — |
| حظر / إلغاء حظر IP | ✓ | — | — |
| تعديل الإعدادات | ✓ | — | — |

## 7. استخدام الواجهة

### 7.1 لوحة التحكم (`/`)

- **بطاقات KPI:** استخدام المعالج، الذاكرة، القرص، الاتصالات النشطة.
- **رسم الأداء اللحظي:** يتحدث كل 5 ثوانٍ.
- **ملخص الأمان:** عدد الكيانات، الاستجابات، الأحداث، التنبيهات.
- **أحداث أمنية حديثة:** آخر الأحداث المسجلة.
- **إجراءات سريعة:** تحديث البيانات، تقرير أمني (حسب الصلاحية).

زر **تحديث** يعيد جلب البيانات من API فوراً.

### 7.2 التحليلات (`/analytics`)

- صحة النظام وكفاءة الموارد.
- رسوم: اتجاهات الأداء، توزيع الموارد، أحداث الأمان، تحليل المخاطر.
- جدول **Top Processes** — أكثر العمليات استهلاكاً للمعالج.

### 7.3 إدارة التهديدات (`/threats`)

| الإجراء | كيفية التنفيذ |
|---------|----------------|
| عرض التهديدات النشطة | تلقائي عند فتح الصفحة |
| حظر IP | زر **حظر IP** → أدخل العنوان والسبب → **Block IP** (مدير فقط) |
| إلغاء الحظر | زر **إلغاء الحظر** في جدول العناوين المحظورة |
| تحديث البيانات | زر **تحديث المسح** |

> حظر IP يُسجَّل في قاعدة البيانات. إذا كان `CDS_FIREWALL_ENABLED=true` يحاول أيضاً تطبيق قاعدة على جدار الحماية (يتطلب صلاحيات مسؤول على Windows/Linux).

### 7.4 الإعدادات (`/settings`)

- معلومات الجلسة الحالية والصلاحيات.
- تبويب **Metric Contract:** مرجع المقاييس ومصادرها.
- المدير فقط يرى زر **حفظ** للإعدادات العامة.

### 7.5 مؤشرات الواجهة

| العنصر | المعنى |
|--------|--------|
| شريط جودة البيانات (أعلى) | حداثة البيانات ومصدرها |
| شارة الدور | admin / analyst / viewer |
| **متصل / غير متصل** (أسفل يسار) | حالة WebSocket للبث اللحظي |
| القائمة الجانبية (موبايل) | زر القائمة الدائري أسفل اليمين |

### 7.6 تسجيل الخروج

من الشريط الجانبي: **تسجيل الخروج**.

## 8. استخدام واجهات API مباشرة

العنوان الأساسي: `http://127.0.0.1:8080`

### 8.1 تسجيل الدخول (جلسة)

```bash
curl -X POST http://127.0.0.1:8080/api/login \
  -c cookies.txt \
  -F "username=admin" \
  -F "password=changeme"
```

### 8.2 قراءة المقاييس

```bash
curl -b cookies.txt http://127.0.0.1:8080/api/system-metrics
```

### 8.3 تحليل كيان

```bash
curl -b cookies.txt -X POST http://127.0.0.1:8080/api/analyze-entity \
  -H "Content-Type: application/json" \
  -d '{
    "entity_id": "workstation_01",
    "entity_data": {
      "connection_rate": 0.8,
      "request_rate": 0.7,
      "failed_auth_count": 12,
      "total_auth_count": 20,
      "unique_ports": 0.6,
      "sensitive_access_count": 0.4,
      "source_ip": "192.168.1.50"
    }
  }'
```

### 8.4 حظر IP (مدير فقط)

```bash
curl -b cookies.txt -X POST http://127.0.0.1:8080/api/block-ip \
  -H "Content-Type: application/json" \
  -d '{"ip_address": "10.0.0.99", "reason": "manual_block"}'
```

### 8.5 وثائق API التفاعلية

افتح: http://127.0.0.1:8080/docs (Swagger UI من FastAPI).

## 11. سيناريوهات استخدام شائعة

### سيناريو 1: مراقبة يومية (مشاهد)

1. سجّل دخول بحساب `viewer`.
2. افتح **لوحة التحكم** لمراقبة CPU والذاكرة.
3. راجع **التحليلات** لاتجاهات الأداء.
4. لا يمكنك حظر IPs أو تحليل كيانات.

### سيناريو 2: تحليل تهديد (محلل)

1. سجّل دخول بحساب `analyst`.
2. من **إدارة التهديدات** راجع التهديدات النشطة.
3. استخدم تحليل الكيان عبر API أو من لوحة التحكم (إن وُجد الزر).
4. أنشئ **تقرير أمني** من لوحة التحكم (يُحمَّل كملف JSON).

### سيناريو 3: حظر عنوان مشبوه (مدير)

1. سجّل دخول بحساب `admin`.
2. اذهب إلى **إدارة التهديدات**.
3. اضغط **حظر IP** وأدخل العنوان.
4. تحقق من ظهوره في جدول **Blocked IP Addresses**.

### سيناريو 4: تطوير بدون Docker

1. اترك `CDS_DATABASE_URL` فارغاً في `.env`.
2. شغّل `python -m backend.main` و`npm run dev`.
3. البيانات تُخزَّن في `data/cyber_defense.db`.

## 12. استكشاف الأخطاء

| المشكلة | الحل المحتمل |
|---------|----------------|
| `401 Unauthorized` على API | سجّل دخولاً أولاً عبر `/api/login` أو الواجهة |
| `403 Forbidden` | حسابك لا يملك الصلاحية — جرّب حساب مدير |
| الواجهة لا تتصل بالـ API | تأكد أن Backend يعمل على 8080 وأن `npm run dev` يعمل |
| خطأ اتصال PostgreSQL | شغّل `docker compose up -d postgres` أو راجع `CDS_DATABASE_URL` |
| Docker غير موجود | اترك `CDS_DATABASE_URL` فارغاً لاستخدام SQLite |
| WebSocket «غير متصل» | طبيعي إن لم تكن مسجّل دخولاً؛ أعد تحميل الصفحة بعد الدخول |
| حظر IP لا يُطبَّق على النظام | عطّل `CDS_FIREWALL_ENABLED=false` أو شغّل كمسؤول |
| منفذ 8080 مشغول | غيّر `CDS_PORT` في `.env` وحدّث بروكسي Vite إن لزم |
